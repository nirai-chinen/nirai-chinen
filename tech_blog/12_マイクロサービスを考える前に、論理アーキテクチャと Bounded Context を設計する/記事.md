# マイクロサービスを考える前に、論理アーキテクチャと Bounded Context を設計する

## 原則：最初から物理アーキテクチャを考えない

物理アーキテクチャとは、サービス、UI、データベース、デプロイ単位など、実際の物理的な構成のこと
いきなり「これはマイクロサービスにする」「これはDBを分ける」と考えると、システムの機能がどこにあり、どのように分割されているかが不明瞭になりやすい。

まずは論理アーキテクチャで、

システムが何をするのか
機能がどう分かれているのか
機能同士がどう相互作用するのか

を整理する。その後で、モノリスにするのか、マイクロサービスにするのか、イベント駆動にするのかといった物理設計を考える。

論理コンポーネントとは、システム内の主要なビジネス機能を実装する単位である。
たとえば受注システムなら、次のようなものが論理コンポーネントになる。

Inventory Management
Order Shipment
Payment Processing
Order Fulfillment
Customer Notification

## 原則：論理コンポーネントの設計は、一度で完成させるものではない

初期の候補コンポーネントを作り、要件やユーザーストーリーを割り当て、責務を分析し、アーキテクチャ特性を考慮しながら、継続的に再構成していく。

## Step: 初期の論理コンポーネントを見つける代表的な方法

### アクター／アクションアプローチ

システムに関わるアクターと、そのアクターが行う主要アクションからコンポーネントを見つける方法。

たとえば受注システムなら、

顧客：商品検索、注文、注文キャンセル、顧客登録
オーダーパッカー：箱サイズ選択、出荷準備完了の記録
システム：在庫調整、支払い適用、補充発注

のように整理し、それぞれのアクションに対応するコンポーネントを考える。

#### アンチパターン：エンティティの罠

避けるべきアンチパターンとして、エンティティをそのままコンポーネントにする方法が紹介されている。

たとえば、

Customer Manager
Item Manager
Order Manager

のように、Customer、Item、Order といった名詞をそのままコンポーネント化する設計である。

これは問題が多い。

第一に、名前が曖昧で責務がわからない。
Order Manager と言われても、「注文を管理する」以上の具体性がない。

第二に、関連する機能が何でも詰め込まれる。
Order Manager に注文検証、注文受付、履歴、履行、出荷、追跡などが入り、巨大な「なんでもあり」コンポーネントになってしまう。

第三に、保守・テスト・デプロイが難しくなる。

Manager、Controller、Handler、Processor、Engine などの曖昧な接尾辞は、エンティティの罠に陥っている兆候になりやすい。

## Step: ユーザーストーリーを割り当てて責務を明確にする

初期コンポーネントは、まだ空のバケツのようなもの。
そこにユーザーストーリーや要件を割り当てることで、具体的な役割と責務が明確になる。

たとえば、

注文を検証する → Order Placement
箱のサイズを決定する → Order Fulfillment
注文状況をメール通知する → Customer Notification

というように割り当てる。

もし既存のコンポーネントのどれにも自然に属さない要件が出てきた場合、新しいコンポーネントを作るべきサインになる。
メール通知のように複数コンポーネントから使われる処理を各所に重複実装するのではなく、Customer Notification のような独立したコンポーネントとして切り出す。

## Step: コンポーネントが責務を持ちすぎていないかを確認する

6. 役割と責務の分析

コンポーネントに要件を割り当てたら、そのコンポーネントが責務を持ちすぎていないかを確認する。

重要な観点は凝集度である。
凝集度とは、コンポーネント内の操作や責務がどれだけ相互に関連しているかを表す。

たとえば Order Placement に次のような責務があるとする。

注文を検証する
カートを表示する
配送先住所を決定する
支払い情報を収集する
注文IDを生成する
支払いを適用する
在庫を調整する
注文概要をメール送信する

これらはすべて「注文」に関係しているが、支払い、在庫、メール通知まで Order Placement に入れると責務過多になる。

その場合は、

Order Placement
Payment Processing
Inventory Management
Customer Notification

のように分割した方がよい。

責務文を書いたときに、「および」「さらに」「また」などの接続詞が多くなる場合、そのコンポーネントは責務を持ちすぎている可能性が高い。

## Step: コンポーネント分割は、機能だけで決まるわけではない。

スケーラビリティ、信頼性、可用性、耐障害性、弾力性、アジリティなどのアーキテクチャ特性も影響する。

たとえば、同じ「ユーザー入力を処理する」機能でも、

数百人・数千人の同時ユーザーを扱う部分
数人しか使わない管理者向け部分

では、必要なスケーラビリティや可用性が異なる。
その場合、機能的には同じに見えても、別コンポーネントに分ける判断が妥当になる。

つまり、論理アーキテクチャを設計する前に、重要なアーキテクチャ特性を把握しておく必要がある。

## 区切られた文脈とは

区切られた文脈とは、あるモデルと言葉の意味が一貫して通用する範囲です。

「見込み客」の例では、次のように文脈を分けます。

販売促進の文脈：見込み客 = 連絡先を伴う関心の通知
営業の文脈：見込み客 = 営業活動全体を管理する対象

重要なのは、同じ「見込み客」という言葉を組織全体で無理に一つの意味に統一することではありません。文脈の内側で意味が一貫していればよいのです。

## 区切られた文脈は「小ささ」で決めない

区切られた文脈（Bounded Context）の適切な大きさは、最初から機械的に決められるものではありません。

重要なのは、先にモデルを理解し、その後で境界を決めることです。単に「小さいサービスがよい」という考えで分割すると、後になって複数の文脈にまたがる変更が頻発し、調整コストや技術的負債が増えます。

初期段階では次の方針が有効です。

- 最初は広めの区切られた文脈として設計する
- 業務理解が進み、安定した境界が見えてから小さく分割する
- 中核領域と頻繁に連携する領域は、初期には同じ文脈に含めることも検討する

物理的に分離したサービスを後から統合するより、論理的な境界を後で分割するほうが容易だからです。

## マイクロサービスの境界の上限、下限

ドメイン駆動設計（DDD）は、適切なマイクロサービス境界を考えるために有効です。特に重要なのは、次の三つの境界です。

DDDの概念	マイクロサービス境界との関係
区切られた文脈	有効な境界の最も広い側を示す　上限
集約	有効な境界の最も狭い側を示す　下限

## BC内のコードの共通化はOK

ユビキタス言語が同じなので大きな問題はないだろう。
マルチモジュールと掛け合わせて、コードは共通しているが、インフラ側でマイクロサービスが分かれている。とすることができる。
これにより複雑性をある程度低減可能になるのではないか

筆者の所感

## BCでの分割が先、技術での分割は最後

言語要因や非同期処理の必要性など、技術要因でマイクロサービスを分けることもあるだろう。
その場合、BCでの分割を先に、技術での分割を後に考えなければ、BCが混ざり分散した泥団子になると思う。

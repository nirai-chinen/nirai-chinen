# c-CRAB 論文を読むと、AIにコードレビューの全体の40%、特定領域は任せてもいいのではと思った

https://arxiv.org/abs/2603.23448v1

AIがコードを書く量は急増しています。すると次に詰まるのは実装ではなく、レビューです。
最近の論文 **Code Review Agent Benchmark (c-CRAB)** は、この状況に対して「AIレビューをどこまで信用してよいのか」を考えるうえで、かなり重要な材料を出しています。 

この記事では、この論文をベースに次の問いに答えます。

* 何%くらいAIにレビューを任せてよいのか
* どのレビュー領域をAIに任せるべきか
* どこはまだ人間が握るべきか

先に結論を書くと、**2026年3月時点では、AIに任せてよいのは「一次レビューの30〜40%程度」まで**、ただしそれは**領域を限定した場合**です。
**最終承認や設計判断まで含めてAIに丸投げする段階ではまだない**、というのがこの論文から引ける実務的なラインだと思います。これは論文の結果を踏まえた私の実務寄りの解釈です。根拠となる実測では、評価対象ツールの総合 pass rate は **20.1%〜32.1%**、4ツールを合わせた和集合でも **41.5%** にとどまっています。 

## この論文は何を測っているのか

c-CRAB の面白いところは、レビュー品質を「人間のレビュー文にどれだけ似ているか」で測っていない点です。
論文では、PRに対して人間が実際に指摘した問題をもとに**実行可能なテスト**を作り、そのテストを通せるレビューを出せたかで評価しています。つまり、

1. PRを入力する
2. レビューAIがコメントを出す
3. 別のコーディングエージェントがそのコメントをもとに修正する
4. 修正版が人間レビュー由来のテストを通るかを見る

という流れです。 

この設計はかなり重要です。
「レビューコメントがもっともらしい」ことではなく、**実際に修正につながる問題発見ができているか**を見ているからです。coding agent時代に欲しいのは、きれいなコメントではなく、マージ前に危ない点を拾えるレビューです。 

## 論文の結果をざっくり読む

論文では Claude Code、Codex、Devin、PR-Agent を評価しています。総合 pass rate は以下でした。 

* Claude Code: **32.1%**
* Devin: **24.8%**
* PR-Agent: **23.1%**
* Codex: **20.1%**

しかも4ツール全部を合わせても、カバーできたのは **234件中97件、41.5%** です。
つまり、「AIレビューを全部乗せしても、人間が見る論点の半分以下しか拾えていない」というのが、この論文の現在地です。 

ここで大事なのは、この数値をそのまま「AIレビューの精度」と読むべきではないことです。論文自身も、これは「AIが無価値」という意味ではなく、**人間とAIが見ている論点が違う**可能性を示していると述べています。 automated tools は humans の代替ではなく、**complement** として見るべきだ、というのが著者の立場です。 

## では、何%AIに任せてよいのか

ここからは論文の結果を、実務判断に落とします。

私の結論はこうです。

> **AIに任せてよいのは、一次レビュー工程の30〜40%程度。**
> ただし対象は、**局所的で、パッチから推測しやすい論点**に限る。
> **最終ゲートは人間が持つ。**

この「30〜40%」は、論文の **単体最高 32.1%** と **複数ツール和集合 41.5%** を、そのまま運用指針に翻訳したものです。 

もちろん、これは「レビュー時間の40%を削減できる」という意味ではありません。
もっと正確には、**人間が本来やっていたレビュー論点のうち、30〜40%程度はAIに先に洗わせてもよい**、という意味です。

このラインを超えて、たとえば「AIがOKと言ったPRは人間確認なしで通す」運用にすると危ないです。なぜなら論文では、人間レビューが100%通すベンチマークに対し、自動レビューはかなり大きなギャップを残しているからです。 

## AIに任せやすい領域

論文の RQ2 では、レビュー論点をカテゴリ分けして、人間とAIの違いを見ています。
その結果、AIは **robustness** と **functional correctness** 系で比較的高い pass rate を示し、さらに **testing** 系の指摘も人間より多く出す傾向がありました。著者らは、これらはパッチ局所に現れやすく、見つけやすいからだと解釈しています。 

なので、AIに任せやすいのは次の領域です。

### 1. エッジケース・境界条件の漏れ

* null / empty / extreme input
* 例外系の分岐漏れ
* コーナーケースでの挙動不整合

これは論文でいう **robustness** に近い領域です。AIはこの種の論点を比較的よく拾います。 

### 2. 明確な機能的不整合

* 仕様から見て明らかに返り値が怪しい
* 条件分岐が逆
* 回帰バグの可能性
* 実装とテストの意味的不一致

これは **functional correctness** に相当します。局所的に検査しやすいので、一次レビューとしてAIを使いやすいです。 

### 3. テスト不足・弱いアサーション

* 新規分岐に対するテスト不足
* 失敗系テストがない
* flaky / fragile なテスト
* assertion が弱く検知力が足りない

論文では、AIは **testing** への感度が高めです。
「レビュー担当AIにテスト観点を多めに見せる」は、かなり相性がよい運用です。 

### 4. セキュリティ・互換性・エラーハンドリングの一次スクリーニング

論文では、**security / efficiency / compatibility / error handling** に関して、AIが出すコメント比率は人間に比較的近いとされています。 

ここでのポイントは「最終判断を任せる」ではなく、**一次スクリーニングを任せる**ことです。
たとえば次のような使い方です。

* 例外メッセージの欠落や雑さを拾う
* 互換性を壊す変更候補を炙り出す
* 明らかな unsafe path を洗う
* 性能劣化が起きそうなホットパス変更を警告させる

## まだAIに任せにくい領域

逆に、人間が強く持つべき領域も論文ではかなり明確です。

### 1. 設計レビュー

論文では **design** はAIが弱いカテゴリのひとつです。
APIの形、責務の切り方、将来の拡張性、依存方向、他モジュールとの整合性のような論点は、パッチ局所からは読み切れません。 

ここをAIに任せきると、短期的には動くが長期的に崩れる変更を通しやすくなります。

### 2. 保守性・repo固有の流儀

論文では **maintainability** の pass rate は **7.9%〜27.0%** とかなり低く、著者は理由として **repository-specific context** の不足を挙げています。 

これは実務感とも一致します。AIは一般論ではレビューできても、

* このrepoではそのユーティリティを使う
* この層では logging の書き方が決まっている
* 同系機能はこの抽象に寄せる
* チームではこの例外ポリシーに統一する

といった**ローカル慣習**に弱いです。
この領域は、まだ人間が持つ価値が大きいです。 

### 3. ドキュメント・命名・意図の整合

論文でも **documentation** は低 pass rate 側です。
README、コメント、サンプルコード、migration note、命名の一貫性、利用者への伝わり方は、単なる局所コード解析では片づきません。 

### 4. 「この変更を入れるべきか」の判断

AIは「このコードに問題があるか」はある程度見ても、
**そもそもこの方向で実装すべきか** までは、まだ安定して判断できません。

* 今は対症療法でよいのか
* 今この抽象化を導入すべきか
* リリースタイミング的にスコープを絞るべきか
* この機能はプロダクト方針として正しいか

この種の判断は、論文のベンチマーク対象の外でもあり、実務上はなおさら人間が担うべきです。 

## 実務でのおすすめ分業

この論文から自然に出てくる分業は、こうです。

### AIの担当

* robustness
* functional correctness
* testing
* security / compatibility / error handling の一次洗い出し

### 人間の担当

* design
* maintainability
* documentation
* repo固有の規約・文化・方針
* 最終承認

これは論文の “Automated review tools should be viewed as complements to human reviewers rather than replacements” という actionable insight とほぼ一致しています。 

## 運用に落とすならどうするか

おすすめは、AIを**一次レビューア**に固定することです。

たとえばフローはこんな感じです。

1. PR作成時にAIレビューを自動起動する
2. AIには「robustness / correctness / testing / security」を優先させる
3. 人間レビューアは「設計・保守性・ドキュメント・整合性」に集中する
4. AIコメントは fail/pass の根拠が薄いものを自動で折りたたむ
5. 最終 approve は人間のみ

この形なら、AIの強い領域だけ使いつつ、弱い領域を人間が補えます。
論文でも、AIは人間より多くコメントする一方で、有効性は低いと報告されているので、**量はAI、責任は人間** の分担が妥当です。 

## じゃあ「何%任せるか」の最終回答

この記事の最終回答を一文で書くとこうです。

> **2026年時点では、AIに任せてよいのはレビュー全体の30〜40%まで。**
> **ただし、その中身は robustness / correctness / testing / 一部の security・compatibility・error handling に限定する。**
> **design・maintainability・documentation・最終承認は人間が握るべき。**

より厳密に言うと、
**「AIがレビュー全体の30〜40%を自律的に完了する」ではなく、
「人間が本来見るべき論点のうち30〜40%を、一次スクリーニングとしてAIに先出しさせる」**
くらいがちょうどよいです。これは c-CRAB の結果に整合的です。 

## おわりに

coding agent時代に一番変わるのは、「AIがコードを書くこと」そのものではなく、
**人間の主戦場が実装からレビュー設計へ移ること**だと思います。

AIレビューはもう使えます。
ただし、**万能レビューアとしてではなく、局所バグ検出に強い一次レビューアとして**です。

この論文は、その線引きをかなりはっきり見せてくれます。
置き換えではなく分業。
少なくとも今は、そこが一番現実的です。
